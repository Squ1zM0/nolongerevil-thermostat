## Contributing to Documentation

Contributions to improve documentation are welcome! When contributing:

### Documentation Standards

- Use clear, concise language
- Include examples where helpful
- Keep formatting consistent
- Update table of contents when adding sections
- Test all code examples
- Add screenshots for UI changes (when applicable)

### Markdown Style Guide

- Use ATX-style headers (`#`, `##`, `###`)
- Use fenced code blocks with language identifiers
- Use tables for structured data
- Include links to related documentation
- Keep line length reasonable (80-100 characters when possible)

### What to Document

**User-Facing:**
- Installation steps
- Configuration options
- Troubleshooting procedures
- Common use cases
- Best practices

**Developer-Facing:**
- Technical specifications
- API or configuration formats
- Design decisions and rationale
- Implementation notes
- Testing procedures

## Document Structure

### README Files
Should include:
- Brief overview/introduction
- Quick start guide
- Links to detailed documentation
- Common use cases
- Support/help resources

### How-To Guides
Should include:
- Clear step-by-step instructions
- Prerequisites
- Expected outcomes
- Troubleshooting common issues
- Examples

### Reference Documentation
Should include:
- Complete specifications
- All options/parameters
- Default values
- Examples for each option
- Related concepts

### FAQ/Troubleshooting
Should include:
- Questions organized by category
- Clear, actionable answers
- Links to related documentation
- When to seek additional help
